3 种架构方案对比

V853 板端：检测模型 + 报警后处理 + IPC 接口

A

单体一体式 简单但不推荐

一个 C 程序包揽全部：视频采集 → NPU 推理 → 报警判定 → 结果输出

Pros

  • 代码量最少，编译部署简单
  • 延迟最低（函数调用，无 IPC 开销）

Cons

  • 报警逻辑改动需重新编译整个程序
  • 调用方无法独立配置区域
  • 代码耦合严重，难以维护
B

引擎 + 后处理分层 推荐

两层分离，通过共享内存传递检测结果，报警引擎独立可配置

Pros

  • 检测模型和报警逻辑独立演进
  • 调用方可动态配置报警区域
  • 模块清晰，方便调试和测试

Cons

  • 多一层共享内存通信
  • 需要定义清晰的数据接口
C

全功能 SDK 过度设计

独立 SDK 库 + 配置管理 + 多种报警策略 + 远程管理协议

Pros

  • 最灵活，支持未来扩展
  • 适合多项目复用

Cons

  • 开发周期长，当前需求用不上
  • V853 资源有限，过于复杂

推荐 B 方案的理由

  1. 检测模型是核心，独立出来后可以单独迭代（换模型、加类别），不碰报警逻辑
  2. 报警引擎独立，调用方可以动态设置/修改 4 点标定区域，符合你说的"接口提供给调用方"
  3. 两层间共享内存传递检测结果，开销极小（< 1ms）
  4. 与 DEPLOYMENT_GUIDE.md 中现有的编译/部署流程完全兼容
